# Purpose

Owns the frontend application source: React routes, UI, domain rules, data loading, persistence, workers, and shared libraries.

## Ownership

- `App.tsx` owns live route composition and much of the mobile UI shell.
- `domain/` owns feed/filter/share/recommendation types and pure business rules.
- `services/` owns catalog loading, detail loading, and external data-source integration.
- `store/` owns local persistence and migrations.
- `workers/` owns background computation paths.
- `assets/`, `lib/`, and `db/` support source-level UI/data helpers.

## Local Contracts

- Read the root AGENTS.md first.
- Keep frontend data consumption compatible with `manhwa_db/db/exports/frontend`.
- Do not make the frontend read backend raw, processed, enrichment, cache, or state files.
- Do not add a blanket no-recognized-tag exclusion unless the user explicitly asks; backend/API tag coverage can differ from the MangaBaka site.
- Installed PWA users may already have IndexedDB state; migrations must be backward-compatible and recoverable.
- The default PWA identity is `Aeon`; preserve an intentionally customized app name in saved user settings.
- Home pager behavior is protected and user-visible: horizontal feed swipe, per-feed vertical scroll, detail back, and route/session restore must keep the stable `9f16d14` behavior baseline.
- Do not reintroduce commit `6b05599` behavior, `HOME_FEED_PREVIEW_TITLES = 18`, delayed route wrappers, hidden-pane vertical restoration, or delayed post-swipe scroll corrections.

## Work Guidance

- Use existing domain helpers and types before adding new state shapes.
- Keep rendering cheap on mobile; avoid expensive blur, image extraction, or large list updates in scroll handlers.
- Route/detail navigation should respond immediately, even if details or recommendations are still loading.
- Home stays mounted behind a title overlay so detail Back reveals the same feed tree. Feeds, Search, and Settings own separate routes and must not keep a hidden Home pager alive behind them. Returning through Home rebuilds from the saved active feed and per-feed scroll state.
- Home owns its pager and per-feed scroll restoration. The global session restorer exclusively owns the last route and must not read or write Home pane positions; Home scroll events must never mutate route state. Initial route restoration completes before normal route persistence starts, so opening at `/` cannot erase a saved non-Home route. During cold startup, the library loader overlays the mounted pager instead of replacing it, and automatic pager/pane scroll events remain non-persistent until initial alignment has completed. A missing or unmounted Home scroller must never be treated as `scrollTop: 0`; only a measured mounted pane may overwrite its saved position.
- Entering or leaving a temporary Home segment preview changes the pager's feed array. Reset pager ownership before that realignment and re-enable scroll-driven warming afterward; otherwise feeds outside the initial render radius can remain permanent skeletons.
- Feeds-page mosaic preparation processes only the selected library's visible feeds and yields between feed queries. It must be cancellable on unmount so background mosaic work can never block bottom-nav input or delay Home paint.
- Catalogue cover cards use native same-document hash anchors for detail routes so hundreds of mounted cards do not subscribe to every bottom-nav route change. Preserve the existing title URL, Back behavior, selection interception, and pre-navigation preparation.
- Opening a shared feed link has no Add preview. Clone it to a new local ID, place logic feeds in LIST/UNSEGMENTED or custom feeds in MY LIST/UNSEGMENTED, open it directly on Home, and replace the import history entry. Broader settings/full imports retain confirmation.
- Feed sharing always offers both the compressed link and a single-feed JSON download. Importing JSON without segment metadata merges regenerated feed copies into LIST/UNSEGMENTED or MY LIST/UNSEGMENTED by feed kind; segment JSON preserves and imports its segment.
- A successful Copy action closes the share drawer and its underlying feed menu, then shows a short app-styled `Link copied` toast. Support LAN-phone previews with a user-gesture `execCommand` fallback when the secure Clipboard API is unavailable; do not use browser alerts.
- LIST owns logic feeds and MY LIST owns fixed-membership custom feeds. Each library has independent segments and UNSEGMENTED placement, while the saved root library order controls their Home sequence. Existing state without library fields migrates to LIST without changing feed IDs or behavior.
- The Feeds page remembers its last selected LIST/MY LIST tab for the browser session. Opening a feed and returning with Back must restore that tab; this page-tab memory must not alter Home library order, the active Home feed, or persistent library contents.
- Custom feeds store at most 500 unique catalogue IDs. Membership is never changed by filters; manual order is fixed, automatic order uses the saved sort rules, and unavailable catalogue IDs remain stored for future recovery/import compatibility.
- `CREATOR FAVOURITES` is a built-in MY LIST segment containing the fixed `MY FAVOURITES` custom feed. It installs additively for fresh and existing libraries, starts visible on Home, and must not replace or duplicate an existing copy with the same stable IDs. The old supplied description `Sorted by Personal Ratings` migrates to `Sorted by the Creator's Ratings` without replacing membership or settings.
- Custom compact settings include title, description, mobile/desktop grid, rank and cover-stat visibility, Completed/Hiatus filters, insertion placement, and non-AniList placement. Advanced settings keep Chapter min/max and Year min/max visible, while an Add-based Additional Stat Ranges editor folds all other filterable stat min/max controls. Rolling date/window controls, Manual/Automatic mode, and existing multi-sort rules remain available. Logic tag/source and content-rule controls are not supported there and must not remain active as hidden filters after save.
- Custom title rearrangement is the only mode allowed to disable horizontal Home swipe. Enter it from the custom title-selection dock and keep its Done confirmation as a small centered bottom pill above main navigation, not a full-width surface or header overlay. Use explicit app-style glass drag handles and a readable cover-only drag ghost with no title label. Position the ghost at the pointer immediately on pickup so it never flashes at the viewport origin. The window owns the active pointer stream so moving/re-rendering a card cannot strand the drag. After a deliberate stable hover, move an empty source-sized placeholder to the prospective drop slot; while the pointer is outside the title grid, auto-scroll may continue but the placeholder must not chase passing cards. Do not rerender order continuously for every pointer movement. Keep hidden filtered IDs in place and restore swipe immediately when the mode closes; do not rewrite pager ownership.
- If changing feed settings, custom feeds, folders, or profiles, update migration and serialization paths together.
- Built-in default feeds use the compact settings drawer: grid columns, rank visibility, cover-stat visibility, and optional Completed/Hiatus status filtering only. Leaving both status switches off means all statuses. User-created feeds retain the full advanced editor.
- A user-created feed's explicit empty `excludeTagIds` list means no tag exclusions and must survive normalization and persistence; only legacy feeds missing the field receive default sensitive exclusions.
- Built-in sensitive library display names are canonicalized on startup independently of one-time content installation markers, so shipped naming corrections reach existing saved libraries without replacing their order, visibility, collapse state, or settings.
- Recommendations are currently suspended: do not expose a Recs navigation item or mount its page, and do not fetch/load recommendation-feature data. Keep the backend export contract intact for older installed frontend versions; old recommendation URLs redirect to Home.
- Built-in sensitive feed segments are installed through a versioned merge without replacing saved feeds. Newly shipped built-in feeds are appended to their existing sensitive segment while preserving that segment's collapsed and Home-visibility state. The BL/GL and Smut/Erotica Settings toggles gate their visibility on both Feeds and Home. Their own Home visibility setting then controls whether an eligible sensitive segment joins the normal Home sequence; they start hidden. A temporary preview of one closes as soon as its Settings toggle is off. Newly created feeds always enter UNSEGMENTED, never a built-in segment.
- Double-tapping the Home navigation item clears any temporary segment preview and sends one consumable reset command to Home, which resets to the first currently visible feed at its top. It must work even when Home is already active and must never replay after later feed-card navigation.
- Search should remain stable while typing and deleting; debounce expensive work instead of blocking input. Search covers display/source/native/romanized titles, every stored title alias, and author/artist names. Any order of complete query words must match before fuzzy fallback, then Fuse relevance order is preserved after sensitive-content filtering. Search displays up to 120 results. The Smut/Erotica search toggle includes titles identified either by adult tags or by `erotica`/`pornographic` content ratings, because tag coverage is not complete for every title.
- Mobile is the primary interface. Desktop presentation must use desktop-only media queries and must not alter phone layout, navigation geometry, Home pager behavior, or touch interactions below the desktop breakpoint.
- Disable browser/device text autosizing at the document root so Android font inflation cannot overflow fixed mobile controls, pills, cards, or navigation geometry; component font sizes remain app-owned.
- Detail loading must try the active data source first and then configured frontend-data fallbacks. A missing Pages detail endpoint must not leave title opening dependent on repeated failed requests.
- First-run library preparation uses a contained app-owned animation: a 20% visual starting baseline, real verified chunk-byte progress through the middle, and a smooth 2–3 second visual completion during saving. Keep the live sync status visible and never use a top-edge loading line.
- Desktop uses cheap translucent fills instead of large blur/filter compositor layers; the mobile glass treatment remains unchanged.
- Desktop is production-preview tested separately from mobile: use a wide 1360px maximum layout, reserve Home side gutters for previous/next feed controls, and use separately saved 6/7/8 desktop grid settings without changing the phone grid. Size title/rank/cover-stat text for those columns. None of these rules may apply below the 768px desktop breakpoint.
- Double-tapping a logic Home header card opens local feed search. Double-tapping a custom Home header card opens only Search in feed and Add to feed. Rearrange is entered from the custom feed's long-press title-selection dock beside Remove. The title button's existing double-tap Feed Settings shortcut remains unchanged. Local search preserves original feed ranks.
- Long-pressing a logic-feed or Global Search title starts cross-feed collection into one or more custom feeds. Long-pressing inside a custom feed starts removal only for that feed. Selection remains transient and normal Home swipe stays enabled; only custom rearrangement locks it. Browser/phone Back must cancel active title selection on the same page before normal route history can run. Selection styling belongs only to the cover shell and must not tint or outline title text.
- Title selection uses per-title subscriptions rather than passing the complete selected-ID set through every mounted card. A changed selection may rerender only that cover and the selection dock; keep the deliberate long-press activation at 320ms and do not regress it into a whole-Home render.
- The title-selection dock sizes to its controls instead of spanning an empty full-width strip. Deselecting the final title quietly cancels selection mode and removes the dock.
- In custom Add mode, unselected results have no selection badge. Selected-title badges are compact 16px checks aligned inside the cover's top-right corner. Use a search-semantic input with password/autofill suppression and the same external app-owned X control as local feed search, not a native cross inside the input; it clears a non-empty query first and closes Add mode when empty. Browser/phone Back follows the same clear-then-close behavior, so no separate Done button is shown. The selection dock must be hidden while its MY LIST destination drawer is open so the two surfaces never stack.
- In Feeds, tapping a segment name or the empty colored segment shell toggles that segment's block. Nested feed cards and action controls must not trigger the shell toggle. Its adjacent circular control opens rename mode instead; UNSEGMENTED remains collapsible but cannot be renamed.
- Feed-card action menus must open inward from the viewport edge. Determine their left/right anchor from the card's rendered position so narrow mobile grid cards never clip the menu off-screen.
- Feeds groups its uppercase page heading and creation/edit controls inside one neutral smoked-glass toolbar. The following neutral glass instruction strip is one line by default and toggles its full text on tap; neither top surface competes with segment hues. Segment rows show drag, name/count, and Home visibility by default. The page-level edit toggle reveals rename, download, and delete actions; keep those utility actions hidden outside edit mode.
- Segment shells use only the frozen three-color palettes sampled from the approved Home headers (Top 1% Trending, Trending Deep Cut, Recently Completed, Trending Shounen, Discover Mainstream/Upcoming/Deep Cut, Overhyped, and Rising Underdogs/Upcoming). Built-ins have fixed assignments; custom segment IDs deterministically reuse that set. Segment names use compact normal-weight app text inside the dark Home-header-style title pill. Keep this pre-baked and cheap: no cover sampling, image decoding, runtime blur, or order-dependent colors on the Feeds page.
- Search and Settings use the same neutral smoked-glass page headers as Feeds. Search places its title grid on one shared translucent result surface, and Settings groups each section as one coherent glass band; preserve existing controls and behavior while avoiding costly live blur. Global Search history stores up to 99 titles actually opened from Search, newest first, and renders them as the same three-column title grid instead of storing query text.
- Learn includes one compact Gestures card documenting every active double-tap shortcut: Home reset, Home feed-title settings, and Home header-card local search.
- Learn explains that the Feeds eye control hides a segment only from Home, while sensitive segments additionally require their matching Settings toggle.
